# Claude Code Protective Hooks Prompt

Create 3 global Claude Code protective hooks in ~/.claude/hooks/ for users running bypassPermissions mode. If ~/.claude/hooks folders do not exist, create them first. Ensure the file location is correct for Mac / Windows as per the operating system.

All scripts must:
- set -euo pipefail, read stdin with INPUT=$(cat), extract fields with jq -r
- On match: print {"decision":"block","reason":"<message>"} to stderr, exit 2
- No match: exit 0 silently
- Use the actual resolved home path (not ~) everywhere

---

## 1. block-destructive-git.sh — PreToolUse / Bash matcher

Extract jq -r '.tool_input.command // empty'. Block with paired reason strings:

- `git\s+push\b.*(\s--force|\s-f)(\s|$)` — use --force-with-lease instead
- `git\s+reset\s+--hard` — permanently discards uncommitted changes
- `git\s+clean\s+.*-[a-zA-Z]*f` — permanently deletes untracked files
- `git\s+checkout\s+--\s+\.` — permanently discards ALL working-tree changes (note: the `\.` must match a literal dot only, not any character; double-escape in the grep pattern)
- `git\s+branch\s+-D\b` — irreversible branch deletion
- `git\s+stash\s+drop` — permanently discards stashed changes
- `git\s+push\s+.*--mirror` — overwrites entire remote repository
- .env git-commit check: block only when BOTH conditions are true: (a) command matches `git\s+(add|commit)` AND (b) .env appears as a standalone word — `(^|\s)\.env(\s|$)`. This avoids false-positives from exclusion pathspecs like ':!.env' where .env appears inside a larger token.

## 2. block-dangerous-bash.sh — PreToolUse / Bash matcher

Use grep -qiE. Extract same command field. Block:

### rm -rf — tiered logic (evaluate in this exact order):

1. **Allow safe build/cache dirs:** If the rm -rf target matches the safe allowlist, SKIP (do not block):
   `(node_modules|dist|\.next|\.turbo|build|coverage|\.cache|tmp|__pycache__|\.parcel-cache|\.vite|\.output|\.nuxt|\.svelte-kit|out|\.sass-cache|\.eslintcache)`
   Match these as the final path component so `rm -rf ./src/node_modules` is also allowed.

2. **Block known-dangerous targets:** Block rm -rf (or rm -r) when the target is any of:
   - `/` or `/*` (root filesystem)
   - `~` or `$HOME` or the literal resolved home path (entire home directory)
   - `.git` (repository history)
   - `/etc`, `/usr`, `/var`, `/boot`, `/sys`, `/proc` (system directories)
   
   Reason: "rm -rf targeting <target> is catastrophically destructive"

3. **All other rm -rf:** ALLOW. Claude Code routinely needs to delete project subdirectories during refactoring, and blocking all relative-path rm -rf would hamstring normal development workflows.

### rm -f absolute paths — narrowed scope:

- `rm\s+-[a-zA-Z]*f\s+/(etc|usr|var|boot|sys|proc|bin|sbin|lib)/` — forced deletion of system files; block with reason "forced deletion of system directory files"
- Single-file rm -f at project absolute paths (e.g. /Users/darren/project/old-file.js) is ALLOWED — this is routine.

### Destructive DB commands:

- `\bDROP\s+TABLE\b`, `\bDROP\s+DATABASE\b`, `\bTRUNCATE\b` — destructive DB commands (use \b word boundaries on all patterns)

### Process killing:

- `kill\s+-9\s+(-1|0)\b` and `\bkillall\b` — kill-all-processes only; do NOT match `kill -9 <pid>` (targeting a specific PID is routine and legitimate)

### Permissions:

- `chmod\s+(-R|--recursive).*777|chmod\s+777.*(-R|--recursive)` — recursive world-writable only

### Filesystem destruction:

- `\bmkfs\b` — formats a filesystem
- `dd\s+if=/dev/(zero|urandom)` — overwrites disk/partitions

## 3. block-secrets.sh — PreToolUse / Edit|Write matcher

Extract FILE from `.tool_input.file_path` and CONTENT from `.tool_input.new_string // .tool_input.content`.

### File path check:

Block if FILE matches `(^|/)\.env(\.|$)` — BUT exclude template files: if FILE also matches `\.env\.(example|template|sample|defaults)$`, SKIP (do not block). These files contain placeholder values and are meant to be committed.

### Content secret patterns (case-insensitive):

Block if CONTENT matches any of:

- `BEGIN (RSA |EC |DSA |OPENSSH )?PRIVATE KEY` — private key material
- `aws_secret_access_key` — AWS secret key identifier
- `AKIA[0-9A-Z]{16}` — AWS access key ID
- `ghp_[A-Za-z0-9]{36}` — GitHub personal access token
- `sk-[A-Za-z0-9]{48}` — OpenAI API key
- `xoxb-[A-Za-z0-9-]+` — Slack bot token
- `xoxp-[A-Za-z0-9-]+` — Slack user token
- `sk_live_[A-Za-z0-9]+` — Stripe live secret key
- `SG\.[A-Za-z0-9_-]{22}\.[A-Za-z0-9_-]{43}` — SendGrid API key

Note: Do NOT block `-----BEGIN CERTIFICATE-----` — public certificates are not secrets. Only block PRIVATE KEY patterns.
